AI 代理尝试的每个操作都会通过三层安全管道。各层独立互补 — 每层捕获不同类别的威胁。

## 第一层：结构执行（墙壁）

主要防线。结构执行让危险操作**不可能发生**，而不是试图检测它们。

组件：

- **能力门控** — 纯函数集合成员检查。没有能力 = 没有操作。
- **污点追踪** — 会话信任级别限制可用能力。
- **操作规范化** — 在评估前解析路径、解码编码、拆分命令链。
- **网关锁定** — 防止完全绕过安全代理。
- **执行沙箱** — 已批准的命令在受限环境中运行（无网络、只读文件系统、ulimits）。
- **出站扫描** — 扫描出站消息中泄露的密钥、PII 和高熵数据。

结构执行无法通过巧妙的提示、社会工程或提示注入来绕过。

## 第二层：语义分析（守卫）

LLM 判断器分析结构上被允许的操作是否在**上下文中异常**。

判断器的职责范围很窄 — 它不会：
- 检测注入模式（污点追踪处理）
- 匹配已知危险命令（规则处理）

它只问一个问题：**"这个结构上被允许的操作，现在看起来是否异常？"**

判断器与规则引擎**并行**运行。如果判断器响应缓慢（>500ms），第一层 + 第三层立即做出决策；判断器的结果用于异步事后审查。

判断器决策：
- `NORMAL` — 操作看起来正常
- `ANOMALOUS` — 操作不寻常 → `REQUIRE_APPROVAL`
- `DANGEROUS`（置信度 ≥ 0.8）→ `DENY`

如果判断器不可用且风险评分超过 0.3，该操作需要审批。

## 第三层：确定性规则（警报）

已知危险的命令模式和路径阻止。快速、零延迟、零歧义。

规则在 YAML 中定义，运行时加载：

```yaml
# 基线规则示例
version: 1
rules:
  - pattern: "rm -rf /"
    action: DENY
    reason: "Recursive delete of root"
  - pattern: "chmod 777"
    action: DENY
    reason: "World-writable permissions"
```

规则始终优先：如果规则说 DENY，无论其他层如何决策，操作都会被拒绝。

## 决策合并

各层决策通过取**最严格**的结果进行合并：

| 条件 | 结果 |
|---|---|
| 规则 DENY | **DENY**（始终） |
| LLM DANGEROUS（置信度 ≥ 0.8） | **DENY** |
| LLM ANOMALOUS | **REQUIRE_APPROVAL** |
| LLM 不可用 + 风险 > 0.3 | **REQUIRE_APPROVAL** |
| 所有层通过 | **ALLOW** |

## 失败安全原则

每次失败都会让系统**更严格**，而非更宽松：

- LLM 宕机 → 需要人工审批
- 配置损坏 → 拒绝启动
- 污点错误 → 假设最高污点
- 规范化失败 → 视为最大危险

## 管道流程

```
Action → Capability Gate → Taint Check → Normalize → ┬─ Rules Engine ──┬→ Merge → Decision
                                                      └─ LLM Judge ─────┘
```

能力门控和污点检查是同步且即时的。规范化为规则引擎和 LLM 判断器的并行评估准备操作。合并器取最严格的结果。
