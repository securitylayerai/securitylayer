テイントレベルは、セッション内のデータの**信頼の出所**を表します。AI エージェントが信頼されていないコンテンツを読み取ると、セッションのテイントレベルが上昇し、明示的にクリアされるまで下がることはありません。

## 6 つのレベル

信頼度の高い順に並べています:

| レベル | 値 | 説明 |
|---|---|---|
| `OWNER` | 1 | 認証済みオーナーから直接 |
| `TRUSTED_CONTACT` | 2 | 許可リストに登録されたコンタクトから |
| `UNTRUSTED_HUMAN` | 3 | 不明または未確認の人物から |
| `WEB_CONTENT` | 4 | Web フェッチ、ブラウザ、メールから |
| `SKILL_GENERATED` | 5 | スキルの指示によって生成 |
| `MEMORY_REPLAY` | 6 | 永続メモリから読み込み |

## 主な特性

### テイントは上昇のみ

セッションレベルのテイントは**単調増加**です。セッションが `OWNER` で開始し、Web コンテンツを読み取ると、`WEB_CONTENT` に上昇します。自動的に下がることはありません。

```
OWNER → 信頼されていないファイルを読み取り → UNTRUSTED_HUMAN → URL をフェッチ → WEB_CONTENT
```

### テイントがケイパビリティを制限する

コロン構文を使用して、ケイパビリティに最低テイントレベルを要求できます:

```yaml
capabilities:
  - exec:trusted        # テイントが TRUSTED_CONTACT 以下の場合のみ許可
  - file.write:owner    # テイントが OWNER の場合のみ許可
  - web_fetch           # どのテイントレベルでも許可
```

セッションのテイントがケイパビリティの閾値を超えると、ベースケイパビリティが付与されていてもアクションは拒否されます。

### テイントのクリア

テイントは明示的な人間の操作でのみクリアできます:

```bash
sl taint clear
```

デーモンモード（プロキシ）では、新しいセッション開始時にテイントがリセットされます。CLI モードでは、プロジェクト信頼ルールに基づいて評価ごとにテイントが追跡されます。

## プロジェクトベースのテイント

Security Layer はプロジェクトディレクトリに基づいて初期テイントを割り当てます:

```yaml
# ~/.securitylayer/projects.yaml
trust_rules:
  - path: "~/Dev/Personal/**"    # OWNER
  - path: "~/Dev/Work/**"        # TRUSTED_CONTACT
  - path: "/tmp/**"              # WEB_CONTENT
  - path: "~/Downloads/**"       # WEB_CONTENT
default: UNTRUSTED_HUMAN
```

これは**ファイルをチャネルとした脅威**をモデル化しています。クローンしたリポジトリには、README、`.cursorrules`、`CLAUDE.md`、コードコメント、`package.json` の postinstall スクリプト、git フックなどに敵対的な指示が含まれている可能性があります。

## なぜこれが重要か

テイントトラッキングがなければ、悪意のあるファイルを読み取ってコマンドを実行する AI エージェントと、正当な指示に従う AI エージェントを区別できません。テイントトラッキングにより、エージェントのコンテキストが信頼されていないコンテンツの影響を受けていることをシステムが認識し、その操作を適切に制限できます。
