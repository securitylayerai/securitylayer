## withSecurityLayer()

Middleware-Wrapper, der SicherheitsprĂĽfungen zu jeder Funktion hinzufĂĽgt. FĂ¤ngt Aufrufe transparent ab â€” die gewrappte Funktion hat dieselbe Signatur wie das Original.

```typescript

```

**Signatur:**

```typescript

  sl: SecurityLayer,
  fn: (...args: TArgs) => TReturn | Promise<TReturn>,
  tool: string,
  options?: {
    extractParams?: (...args: unknown[]) => Record<string, unknown>;
  },
): (...args: TArgs) => Promise<TReturn>;
```

**Verhalten:**
- `ALLOW` â€” ruft die Originalfunktion auf und gibt ihr Ergebnis zurĂĽck
- `DENY` â€” wirft `SecurityLayerError`, ohne die Funktion aufzurufen
- `REQUIRE_APPROVAL` â€” wartet auf Genehmigung, wirft bei Ablehnung

**Standard-Parameterextraktion:** Wenn das erste Argument ein String ist, wird es auf `{ command: arg }` gemappt. Wenn es ein Objekt ist, wird es direkt als Parameter verwendet.

## Fehlerklassen

Alle SDK-Fehler erweitern `SecurityLayerError`, welches `Error` erweitert.

```typescript
import {
  SecurityLayerError,
  ConfigError,
  InitializationError,
  CheckError,
  ApprovalTimeoutError,
} from "@securitylayerai/sdk";
```

| Fehler | Properties | Wird geworfen wenn |
|---|---|---|
| `SecurityLayerError` | `message` | Basisklasse; wird auch von `withSecurityLayer()` bei DENY geworfen |
| `ConfigError` | `configPath?: string` | Konfigurationsdateien kĂ¶nnen nicht geladen oder geparst werden |
| `InitializationError` | `cause?: unknown` | Security pipeline kann nicht initialisiert werden |
| `CheckError` | `tool?: string` | Aktionsauswertung schlĂ¤gt unerwartet fehl |
| `ApprovalTimeoutError` | `approvalId: string` | Genehmigungsanfrage lĂ¤uft ab |
