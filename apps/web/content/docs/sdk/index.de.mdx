## Warum das SDK verwenden?

Die CLI-Integrationen (Hooks, Shell-Shim) funktionieren ohne Code-Änderungen — operieren aber an der Grenze. Das SDK bietet dir **In-Process-Sicherheit** mit vollem Kontext:

| | Shell-Shim | Hooks | SDK |
|---|---|---|---|
| Fängt exec ab | Ja | Ja | Ja |
| Fängt Dateischreibvorgänge ab | Nein | Ja | Ja |
| Fängt Web-Fetch ab | Nein | Ja | Ja |
| Voller Session-Kontext | Nein | Teilweise | **Ja** |
| Taint-Tracking-Präzision | Grob | Moderat | **Pro Operation** |
| Umgehungsrisiko | PATH-Manipulation | Keines | **Keines (In-Process)** |
| Funktioniert mit jedem Framework | Ja (nur Shell) | Nur Claude Code | **Jedes Framework** |
| Integrationsaufwand | Null | Ein Befehl | Code-Änderungen |

Das SDK ist die richtige Wahl, wenn du eine agentische Anwendung baust und strukturelle Sicherheitsgarantien willst — nicht nur Befehlszeilen-Abfangen.

## Schnellbeispiel

```typescript
import { createSecurityLayer } from "@securitylayerai/sdk";

const sl = await createSecurityLayer();

// Check an action before executing it
const result = await sl.check("exec", { command: "rm -rf /tmp/output" });

if (result.decision === "DENY") {
  console.error(`Blocked: ${result.reason}`);
} else if (result.decision === "REQUIRE_APPROVAL") {
  const approved = await sl.waitForApproval(result.approvalId as string);
  if (!approved) console.error("Approval denied");
}

// Clean up when done
sl.destroy();
```
